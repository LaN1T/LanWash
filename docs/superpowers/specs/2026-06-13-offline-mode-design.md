# Дизайн офлайн-режима

## Контекст

Приложение LanWash активно используется мойщиками и администраторами. В условиях автомойки связь может пропадать (подвал, технические помещения, слабый Wi-Fi). Нужно дать возможность продолжать работу без сети, а при восстановлении — синхронизировать изменения.

## Цель

Обеспечить устойчивую работу приложения при отсутствии интернета на мобильных и десктоп-платформах.

## Scope MVP

### Включается

1. **Локальное хранилище данных**
   - Библиотека: `drift` (SQLite) или `isar`
   - Выбор: `drift` для совместимости с десктопом и сложными запросами
   - Кэшируются:
     - справочники (wash types, services, promos, categories)
     - список мойщиков
     - записи пользователя / его смены
     - актуальное расписание смен на текущую и следующую неделю
     - профиль текущего пользователя

2. **Очередь исходящих действий**
   - При офлайн все мутации сохраняются в очередь:
     - create/update/delete appointment
     - create/update/delete shift
     - update profile / avatar
     - create note
   - Каждое действие имеет id, timestamp, endpoint, payload, retry count
   - Показывается индикатор «ожидает синхронизации»

3. **Детектор состояния сети**
   - `ConnectivityPlus` для определения наличия сети
   - При переходе online автоматический запуск синхронизации
   - Ручная кнопка «Синхронизировать» в профиле / настроках

4. **Оптимистичный UI**
   - Действия отображаются сразу, даже если нет сети
   - Сущности в офлайн помечаются статусом `pending`
   - При конфликте синхронизации показывается диалог выбора (local / server)

### Не включается (в MVP)

- Полная двусторонняя синхронизация всех сущностей
- Автоматическое разрешение сложных конфликтов
- Офлайн-режим для админской аналитики и отчётов
- Push-уведомления при восстановлении синхронизации

## Архитектура

### Слои

```
UI → OfflineRepository → LocalDB (drift)
                 ↓
            SyncService → API
```

### Компоненты

1. **`LocalDatabase`** (drift)
   - Таблицы: `CachedWashType`, `CachedService`, `CachedUser`, `CachedAppointment`, `CachedShift`, `PendingAction`
   - DAO для каждой сущности

2. **`OfflineRepository`**
   - `getWashers()`, `getAppointments()`, `getShifts()` — сначала local, потом обновляет из сети
   - `savePendingAction(...)` — добавляет действие в очередь
   - `getPendingActions()` — возвращает неотправленные

3. **`SyncService`**
   - `sync()` — отправляет pending actions в порядке очереди
   - Обработка конфликтов: если 409 / данные изменились — предлагает выбор
   - После успеха обновляет local DB и удаляет выполненные действия

4. **`ConnectivityMonitor`**
   - Стрим состояния сети
   - При online триггерит `SyncService.sync()` с debounce 2 сек

5. **`ApiService` изменения**
   - Все POST/PUT/DELETE сначала пытаются выполнить online
   - При ошибке сети сохраняют действие в очередь и возвращают локальную копию

## Конфликты

Возникают, если данные на сервере изменились между чтением и отправкой офлайн-действия.

Стратегия:
1. При получении 409 от сервера показываем диалог:
   - «Сохранить мои изменения» (перезаписать сервер)
   - «Отменить, взять серверные» (удалить local pending)
   - «Показать различия» (для сложных сущностей)
2. Для простых действий (create) — retry с новым id

## UI/UX

- Индикатор офлайн-статуса в AppBar / шелле
- Snackbar при переходе online/offline
- Иконка «⏳» на сущностях с неотправленными изменениями
- Экран «Очередь синхронизации» в профиле: список pending actions с возможностью удалить/отправить

## Безопасность

- Чувствительные данные (FCM токены, пароли) не кэшируются
- Локальная БД шифруется на мобильных устройствах (SQLCipher через drift)

## Тестирование

- Unit-тесты `SyncService`:
  - отправка очереди в правильном порядке
  - обработка 409
  - удаление выполненных actions
- Интеграционные тесты:
  - создание записи в офлайн → синхронизация
  - редактирование смены в офлайн → конфликт

## Критерии приёмки

- [ ] Приложение запускается и показывает закэшированные данные без сети
- [ ] Мутации в офлайн ставятся в очередь
- [ ] При восстановлении сети очередь отправляется автоматически
- [ ] Конфликты показываются пользователю
- [ ] Работает на Windows / macOS / Linux / iOS / Android
- [ ] Все существующие тесты проходят
